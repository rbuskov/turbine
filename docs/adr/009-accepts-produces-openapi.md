# ADR-009: `Accepts` / `Produces` route metadata and OpenAPI contribution

- **Status:** Accepted
- **Date:** 2026-05-03
- **Deciders:** team

## Context

`RouteHandlerBuilder.Accepts<T>(...)` and `Produces<T>(int, ...)` are the two public
extension surfaces by which an endpoint says "this request body / this response
matches the schema at `T.SomeProperty` on this `SchemaConfiguration`." Until now
both extensions attached a placeholder `WithMetadata(/* metadata */)` and the
ASP.NET Core OpenAPI document generated by `Microsoft.AspNetCore.OpenApi` knew
nothing about Turbine schemas.

We needed to land a real implementation that

1. carries enough information from the call site to a transformer at OpenAPI-
   generation time, and
2. converts each Turbine `ISchema` into a JSON-schema component referenced by
   `$ref` from the relevant operation.

A handful of shape decisions had to be locked in together.

## Decision

### Endpoint metadata

`Accepts<T>` / `Produces<T>` attach a single internal record per call:

```csharp
internal sealed record TurbineEndpointSchemaMetadata(
    Type ConfigurationType,
    string SchemaPropertyName,
    EndpointSchemaRole Role,
    int? StatusCode,
    string ContentType);
```

`EndpointSchemaRole` is internal with values `Request` and `Response`.
`ContentType` defaults to `"application/json"` (per project convention — see
`feedback_json_default`). The metadata, the role enum, and the default content
type are all internal because the only consumer in scope is the Turbine OpenAPI
transformer; if a third-party consumer ever needs them they can be promoted
without a breaking change.

### Selector contract

The expression handed to `Accepts<T>` / `Produces<T>` must be a direct property
access on the lambda parameter, optionally wrapped in a Convert/ConvertChecked
unary expression (so `x => (ISchema) x.Foo` and `x => x.Foo` are both
accepted). Anything else — method calls, indexers, constants, chained property
accesses like `x => x.Inner.Foo` — throws `ArgumentException` whose message
names the offending expression, the configuration type, and the expected shape
(`x => x.SomeSchemaProperty`). Chained property accesses are rejected because
the registry is keyed by `(ConfigurationType, PropertyName)`, not by an
arbitrary path.

### Status-code validation

`Produces<T>` rejects status codes outside the HTTP range with
`ArgumentOutOfRangeException`: any code `< 100` or `>= 600` is invalid. This
fails fast at the call site rather than producing an ill-formed OpenAPI
document.

### Component naming

A schema lives in `components/schemas` under a deterministic name derived from
its registry key:

```
{ConfigurationTypeNameMinusTrailing"Schemas"}_{PropertyName}
```

So `PersonnelSchemas.Summary` → `Personnel_Summary`. The `Schemas` suffix is
stripped only if it isn't the entire type name (so a config literally named
`Schemas` yields `Schemas_Foo`, not `_Foo`). The same `(ConfigurationType,
PropertyName)` key always maps to the same component; the component is
generated exactly once even if multiple endpoints reference it, which is the
deduplication contract the task called out.

### Single document transformer

Schema-component materialisation and operation wiring live in a single
`IOpenApiDocumentTransformer` (`TurbineOpenApiDocumentTransformer`). The
alternative — splitting it into an `IOpenApiOperationTransformer` plus an
`IOpenApiDocumentTransformer` — was rejected because the two passes share the
same component cache and the same recursive owned-reference resolver; keeping
them together means one set of state, one walk of the description groups, and
no need for ordering coordination. The transformer is registered via
`IConfigureNamedOptions<OpenApiOptions>` from `AddTurbine`, so call sites keep
the existing two-line shape (`AddOpenApi(); AddTurbine();`).

The transformer reads schemas from `TurbineSchemaRegistry` (populated by
`MapTurbine`). If the registry is missing or unbuilt at transform time the
transformer is a no-op (the host probably doesn't use Turbine in this OpenAPI
document); if the registry exists but is missing a `(ConfigurationType,
PropertyName)` referenced by an endpoint, it throws
`InvalidOperationException` naming both.

Operation metadata that didn't come from Turbine (e.g. `.Produces(204)`,
`.Produces<decimal>()`, default reflection-derived response shapes) is left in
place; the transformer only writes to `requestBody.content[contentType].schema`
and `responses[statusCode].content[contentType].schema` for the specific
`(role, status, content type)` combinations Turbine actually owns.

### Owned-vs-anonymous nested schemas

When an `ObjectSchema`/`ArraySchema`/`OneOfSchema` references another schema
that is itself owned by a `(ConfigurationType, PropertyName)` slot in the
registry, the nested reference is emitted as `$ref` to that component. Truly
anonymous nested schemas (e.g. an `ArraySchema.Items` configured inline that no
configuration property points at) are inlined.

### Microsoft.OpenApi 2.0 quirk

`OpenApiSchema.Minimum` and `OpenApiSchema.ExclusiveMinimum` (and the same for
`Maximum`) share storage in this version of the model — setting one clears the
other. The converter prefers the exclusive form when both are present on the
Turbine `NumericSchema<T>`, because the exclusive bound is the stricter
constraint. This is documented because it is non-obvious and surfaced in tests.

## Consequences

- ✅ `Accepts<T>` / `Produces<T>` now have real semantics and the rendered
  OpenAPI document matches what the endpoints accept and produce.
- ✅ Component reuse across endpoints is automatic and deterministic.
- ✅ The public `RouteHandlerBuilderExtensions` signature is unchanged.
- ⚠️ Chained selector expressions (`x => x.Inner.Foo`) are rejected; if the
  team ever wants nested-property selectors, that's a contract change and
  needs its own ADR.
- ⚠️ `Maximum`/`ExclusiveMaximum` (and `Minimum`/`ExclusiveMinimum`) cannot
  both appear in the OpenAPI output for one schema in this version of
  Microsoft.OpenApi. Acceptable because the JSON Schema 2020-12 spec also
  deprecates the boolean/value-pair form.
- 🔄 If the registry needs to be public for a third-party transformer or
  validator, the metadata record / role enum / namer can be promoted
  alongside it without breaking call sites.

## Alternatives considered

- **Split into operation + document transformers.** Rejected: shared cache
  and shared walk; nothing needs to live at operation scope.
- **Use a custom `IEndpointMetadataProvider` instead of `WithMetadata`.**
  Rejected: `Produces`/`Accepts` are extension methods on `RouteHandlerBuilder`
  — the metadata-provider interface fires for handler delegates, not for the
  extension call. `WithMetadata` is the right hook.
- **Inline every nested schema.** Rejected: blows up the document for any
  reasonable schema graph and breaks deduplication when the same schema is
  reachable from multiple roots.
- **Public `TurbineEndpointSchemaMetadata`.** Deferred until there is an
  in-tree consumer outside the Turbine assembly; promotion later is
  non-breaking.
